Publicar nas Lojas

Guia passo a passo para publicar o BBrain na App Store (iOS) e Google Play (Android).

🍎
App Store
iOS — iPhone e iPad
$99/ano
1
Apple Developer Program
Cadastre-se no programa de desenvolvedor da Apple. Custa $99/ano e é obrigatório para publicar qualquer app.
developer.apple.com/programs →
2
Criar App ID no Apple Developer Portal
No portal, crie um novo App ID com bundle identifier único (ex: io.aproove.bbrain). Ative as capabilities necessárias (Push Notifications, etc.).
3
Criar app no App Store Connect
Acesse App Store Connect, clique em "+" e crie um novo app usando o App ID criado. Escolha a plataforma e o idioma primário.
4
Preparar os assets visuais
Você precisará de:
Ícone do app
1024×1024px, PNG sem transparência
Screenshots iPhone 6.7"
1290×2796px (obrigatório)
Screenshots iPhone 6.1"
1179×2556px
Screenshots iPad
2048×2732px (se suportar iPad)
5
Preencher metadados
No App Store Connect, preencha: nome do app (30 chars), subtítulo (30 chars), descrição (4000 chars), palavras-chave (100 chars), URL de suporte e URL de privacidade. Escolha a categoria primária.
6
Build e upload
Compile o app via React Native ou Flutter para iOS. Archive via Xcode e faça upload pelo Xcode Organizer ou pelo app Transporter (gratuito na Mac App Store).
7
Submeter para review
Selecione o build no App Store Connect e clique em "Submit for Review". O processo leva de 1 a 3 dias úteis na primeira submissão.
8
Precificação
BBrain pode ser publicado como Free com In-App Purchase (para o plano Pro), ou como Paid App. Recomendado: Free + IAP via RevenueCat ou Stripe.
🤖
Google Play
Android
$25 taxa única
1
Google Play Console
Crie uma conta no Google Play Console. A taxa de registro é única ($25) — sem anuidade.
play.google.com/console →
2
Criar aplicativo
No Console, clique em "Criar app", escolha o idioma padrão e o tipo (app ou jogo). Defina se é gratuito ou pago (não pode mudar depois de gratuito para pago).
3
Ficha do app
Preencha: título (50 chars), descrição curta (80 chars), descrição completa (4000 chars). Faça upload de screenshots (mínimo 2), feature graphic (1024×500px) e ícone (512×512px).
4
Política de privacidade
Obrigatória para todos os apps. Deve estar hospedada em uma URL pública. Use um gerador gratuito se necessário.
privacypolicygenerator.info →
5
Build assinado
Compile o APK ou AAB (Android App Bundle — recomendado). Assine com sua keystore. Para Flutter: flutter build appbundle --release. Para React Native: siga o guia oficial.
6
Lançamento em estágios
Recomendado: comece com Teste Interno (qualquer pessoa com link) → Alpha (até 50 usuários) → Beta (grupo maior) → Produção (lançamento público).
7
Review
O primeiro review leva de 1 a 7 dias. Updates subsequentes são mais rápidos (geralmente horas). A Play Store é mais permissiva que a App Store.

Criar Keystore (uma vez)

keytool -genkey -v -keystore bbrain.keystore \ -alias bbrain -keyalg RSA -keysize 2048 \ -validity 10000

⚠️ Guarde o arquivo bbrain.keystore e a senha em local seguro — sem ele não é possível publicar atualizações.

Build com Flutter (recomendado)

flutter build appbundle --release \ --build-name=1.0.0 --build-number=1

Build com React Native

cd android ./gradlew bundleRelease

Configure android/app/build.gradle com as chaves da keystore via variáveis de ambiente.

Upload no Google Play Console

API BASE URL: https://bbrainapp.you
AUTH HEADER:  Authorization: Bearer {token}
═══════════════════════════════════════════

ALL API ENDPOINTS:

[AUTH — no token required]
GET  /api/auth/status
  → { setup_required: bool, authenticated: bool }

POST /api/auth/login
  body: { username, password }
  → { token, username, force_password_change: bool, firstLoginMessage?: string }

POST /api/auth/setup
  body: { username, password }
  → { token, username }

POST /api/auth/change-password  [requires token]
  body: { password }
  → { success: true }

POST /api/auth/logout  [requires token]
  → { success: true }

POST /api/auth/reset-request
  → { success: true }

POST /api/auth/reset
  body: { code, password }
  → { success: true }

[IDEAS — requires token]
GET  /api/ideas?status=nova&source=whatsapp
  → { ideas: Idea[], meta: { total, on_roadmap, archived, from_whatsapp } }

POST /api/ideas
  body: { text, source?, session_id? }
  → { idea: Idea }

PATCH /api/ideas/:id
  body: { status?, tags?, evaluation?, roadmap_phase?, connections? }
  → { idea: Idea }

DELETE /api/ideas/:id
  → { success: true }

Idea object:
{
  id: string,            // "20260320-ABC12"
  text: string,
  source: "web"|"whatsapp"|"api",
  whatsapp_from: string|null,
  created_at: string,    // ISO 8601
  status: "nova"|"em_analise"|"no_roadmap"|"arquivada",
  tags: string[],
  evaluation: string|null,
  roadmap_phase: "backlog"|"planejado"|"em_execucao"|"concluido"|null,
  connections: string[],
  session_id: string|null,
  updated_at: string|null
}

[SESSIONS — requires token]
GET  /api/sessions
  → { sessions: Session[], meta: { total_sessions, total_hours, last_session } }

POST /api/sessions
  body: { location, initial_thoughts? }
  → { session: Session }

PATCH /api/sessions/:id
  body: { ended_at, duration_minutes }
  → { session: Session }

Session object:
{
  id: string,
  started_at: string,
  ended_at: string|null,
  location: string,
  initial_thoughts: string|null,
  duration_minutes: number|null,
  features_worked: string[],
  ideas_captured: string[],
  social_content: string|null
}

[SETTINGS — requires token]
GET  /api/settings
  → { settings: { plan, reminder_frequency, reminder_time, reminder_channels } }

POST /api/settings
  body: { plan?, reminder_frequency?, reminder_time?, reminder_channels? }
  → { settings: {...} }

settings.plan values: "free"|"pro"|"power"
settings.reminder_frequency: ""|"monthly"|"every15days"|"every2days"|"daily"
  - free plan: monthly, every15days only
  - pro plan: + every2days
  - power plan: all (including daily)
settings.reminder_channels: "email"|"whatsapp"|"email,whatsapp"
  - whatsapp only for pro and power plans

[REPORT — requires token]
GET  /api/report/monthly?month=2026-03
  → {
      month, ideas_count, sessions_count, roadmap_count,
      total_duration (minutes), best_day: { date, count }|null,
      ideas: IdeaReport[], sessions: SessionReport[]
    }

[STRIPE — requires token except webhook]
GET  /api/stripe/plans
  → { plans: [{ id, name, price_brl, price_usd, features[] }] }

POST /api/stripe/checkout
  body: { plan: "pro"|"power" }
  → { url: string, session_id: string }

GET  /api/stripe/portal
  → { url: string }

POST /api/stripe/webhook  [Stripe calls this — no auth]
  Stripe events handled: checkout.session.completed, customer.subscription.deleted

[VERSION]
GET /api/version
  → { version, updated_at }

═══════════════════════════════════════════
AUTHENTICATION FLOW (client-side)
═══════════════════════════════════════════
1. On app start: read token from SecureStore
2. Check if token date === today (local expiry)
3. Call GET /api/auth/status with token
4. If authenticated → load main tabs
5. If not → show LoginScreen
6. After login: store { token, tokenDate } in SecureStore
7. On logout: delete from SecureStore, call POST /api/auth/logout

═══════════════════════════════════════════
STRIPE PAYMENT FLOW
═══════════════════════════════════════════
1. User taps "Fazer upgrade" on SettingsScreen
2. App calls POST /api/stripe/checkout with { plan }
3. Server creates Checkout Session, returns { url }
4. App opens url in expo-web-browser
5. User completes payment on Stripe Checkout
6. Stripe calls POST /api/stripe/webhook → server updates plan in Sheets
7. App polls GET /api/settings on return from browser
8. Plan badge updates in UI

For subscription management:
1. User taps "Gerenciar assinatura"
2. App calls GET /api/stripe/portal
3. Opens returned url in expo-web-browser

Plan restrictions (enforced client-side, validated server-side):
  free:  reminder_frequency ∈ { monthly, every15days }; channel = email only
  pro:   + every2days; + whatsapp channel
  power: + daily; all channels; early access to new features

═══════════════════════════════════════════
OFFLINE SUPPORT
═══════════════════════════════════════════

- On API failure: read from cache, show "offline" indicator
- Queue failed POST/PATCH operations and retry on reconnect
- Use NetInfo to detect connectivity

AUTH FLOW
═══════════════════════════════════════════
POST /api/auth/login
  body: { username, password }
  → { token, username, force_password_change }

If force_password_change === true → show ChangePasswordScreen before app

POST /api/auth/change-password
  headers: Authorization: Bearer {token}
  body: { current_password, new_password }
  → { success }

POST /api/auth/logout
  headers: Authorization: Bearer {token}

All authenticated requests must include:
  Authorization: Bearer {token}

Store token in localStorage key "bbrain_token"
Token TTL: compute from login timestamp, expire after 86400s client-side
On 401 → clear localStorage, redirect to login
