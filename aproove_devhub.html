Introdução

Aproove Developer Hub

A API REST do Aproove permite que você integre aprovação de conteúdo, gestão de campanhas e publicação automática em qualquer sistema externo. Construída sobre HTTP com autenticação OAuth 2.0 e respostas em JSON.

ℹ
Base URL: https://api.aproove.app/v1 — Todas as requisições devem usar HTTPS. Requisições HTTP são rejeitadas com status 301.
✓
A API está em v1 estável. Mudanças breaking são anunciadas no changelog com 90 dias de antecedência e mantemos versões antigas por 12 meses.

Quick start

Três passos para fazer sua primeira chamada de API:

1
Gere uma API Key
No portal do gestor → Configurações → Integrações → API Keys. Cada key tem escopos configuráveis. Nunca exponha sua key no frontend.
2
Adicione o header de autorização
Inclua Authorization: Bearer SEU_TOKEN em todas as requisições. Tokens expiram em 24h para OAuth 2.0 e não expiram para API Keys (até revogação manual).
3
Faça sua primeira chamada
bash
# Listar campanhas ativas da sua agência
curl -X GET "https://api.aproove.app/v1/campaigns?status=active" \
  -H "Authorization: Bearer sk_live_xxxxxxxxxxxx" \
  -H "Content-Type: application/json"
json — resposta
{
  "data": [
    {
      "id": "camp_9xkT2m",
      "name": "Lançamento Verão 2026",
      "client_id": "cli_7nPq4r",
      "status": "active",
      "start_date": "2026-01-01",
      "end_date": "2026-03-31",
      "pieces_count": 18,

  "meta": { "total": 6, "page": 1, "per_page": 20 }

Autenticação

Autenticação e segurança

A API do Aproove suporta dois métodos de autenticação. Use OAuth 2.0 para aplicações que atuam em nome de usuários, e API Keys para integrações server-to-server.

API Keys — server-to-server

⚠
API Keys concedem acesso direto à conta. Nunca exponha no frontend, em repositórios públicos ou em logs. Em caso de comprometimento, revogue imediatamente no painel.
http header
Authorization: Bearer sk_live_xxxxxxxxxxxxxxxxxxxx

Prefixos de API Key: sk_live_ para produção, sk_test_ para sandbox. Keys de teste não geram aprovações reais nem disparam publicações.

OAuth 2.0 — em nome de usuário

Use OAuth 2.0 quando sua aplicação precisa agir em nome de um usuário Aproove. O fluxo é o Authorization Code padrão.

oauth flow
# 1. Redirecione o usuário para autorizar
https://aproove.app/oauth/authorize
  ?client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &scope=campaigns:read approvals:write
  &response_type=code
  &state=random_string

# 2. Troque o code por um access_token
curl -X POST "https://api.aproove.app/v1/oauth/token" \
  -d "grant_type=authorization_code" \
  -d "code=CODE_RECEIVED" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_SECRET" \
  -d "redirect_uri=https://yourapp.com/callback"

# 3. Use o access_token nas chamadas (expira em 24h)
# Renove com refresh_token antes de expirar

Escopos e permissões

Cada API Key ou token OAuth tem escopos que definem o que pode ser lido ou modificado. Configure escopos mínimos necessários — princípio do menor privilégio.

campaigns:read
Listar e visualizar campanhas, peças e grade editorial. Não permite criação ou edição.
campaigns:write
Criar, atualizar e arquivar campanhas. Inclui campaigns:read.
pieces:read
Listar peças, versões, copies e status de produção.
pieces:write
Criar peças, fazer upload de artes e atualizar copies.
approvals:read
Consultar status de aprovação, histórico e auditoria.
approvals:write
Registrar aprovações, solicitações de ajuste e reprovações via API.
clients:read
Listar clientes, usuários aprovadores e configurações de conta.
clients:write
Criar e atualizar clientes, gerenciar usuários aprovadores.
reports:read
Acessar métricas de desempenho e relatórios de ROI por campanha.
webhooks:manage
Criar, listar e excluir endpoints de webhook da conta.
publish:write
Disparar publicação via integrações (Mlabs, Etus, Meta). Requer revisão manual de uso.

Webhooks

Configuração de webhooks

Webhooks permitem que o Aproove notifique seu sistema em tempo real quando eventos importantes acontecem — aprovação, publicação, solicitação de ajuste e mais. Muito mais eficiente que polling.

1
Registre seu endpoint
Cadastre uma URL HTTPS pública que receberá os eventos via POST. O endpoint deve responder com 200 OK em até 10 segundos.
2
Escolha os eventos
Selecione quais eventos disparar para aquele endpoint. Você pode ter múltiplos endpoints com eventos diferentes.
3
Verifique a assinatura
Cada request inclui o header X-Aproove-Signature com HMAC-SHA256 calculado com seu webhook secret. Sempre verifique antes de processar.
bash — cadastrar webhook
curl -X POST "https://api.aproove.app/v1/webhooks" \
  -H "Authorization: Bearer sk_live_xxxx" \
  -d '{
    "url": "https://yourdomain.com/aproove/webhook",
    "events": [
      "piece.approved",
      "piece.revision_requested",
      "piece.published",
      "piece.rejected"
    ],
    "description": "Integração com CRM interno"
  }'

Verificação de assinatura

Todo request de webhook inclui X-Aproove-Signature: sha256=HASH. Verifique antes de processar.

node.js
const crypto = require('crypto');

function verifyWebhook(payload, signature, secret) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(payload, 'utf8')
    .digest('hex');

  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(signature)
  );
}

// No seu endpoint Express:
app.post('/aproove/webhook', (req, res) => {
  const sig = req.headers['x-aproove-signature'];
  if (!verifyWebhook(req.rawBody, sig, process.env.WEBHOOK_SECRET))
    return res.status(401).send('Assinatura inválida');

  const event = req.body;
  if (event.event === 'piece.approved') {
    // Sua lógica aqui — ex: atualizar CRM, disparar Mlabs, etc.
  }

  res.status(200).json({ received: true });
});
Guias de integração

Integração com Mlabs

O padrão de integração com Mlabs usa webhooks do Aproove para disparar agendamentos automáticos. Quando uma peça é aprovada, o evento piece.approved chega no seu endpoint e você chama a API do Mlabs para agendar.

1
Configure o webhook no Aproove
Registre seu endpoint em POST /v1/webhooks com o evento piece.approved. Salve o webhook secret gerado.
2
Receba e verifique o evento
No seu servidor, valide a assinatura HMAC e processe o payload com piece_id, scheduled_at e URL do arquivo.
3
Chame a API do Mlabs
Com os dados recebidos, crie o agendamento na API do Mlabs passando rede social, data/hora, imagem e legenda.
node.js — handler completo
// Quando piece.approved chega:
async function handleApproval(event) {
  const { piece_id, scheduled_at } = event.data;

  // 1. Buscar detalhes da peça no Aproove
  const piece = await aprooveGet(`/v1/pieces/${piece_id}`);

  // 2. Agendar via API do Mlabs
  await fetch('https://api.mlabs.com.br/v2/schedule', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${MLABS_TOKEN}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      network: piece.network,
      schedule_date: scheduled_at,
      text: piece.copy,
      media_url: piece.file_url
    })
  });
}

LGPD e segurança de dados

⚠
Dados pessoais via API: O Aproove não retorna números de telefone completos em nenhum endpoint. O campo whatsapp é retornado mascarado (+55119xxxx4321) e o número completo existe apenas internamente, criptografado. Isso é conformidade LGPD — não contorne com tentativas de engenharia reversa.
  • Todos os dados são armazenados em servidores no Brasil (AWS São Paulo — sa-east-1)
  • Comunicação via HTTPS/TLS 1.3 em todos os endpoints
  • API Keys têm escopo mínimo configurável — nunca use permissões além do necessário
  • Logs de acesso à API são retidos por 90 dias e disponíveis via GET /v1/logs
  • Dados de clientes finais (aprovadores) não são acessíveis via API — apenas pela conta proprietária
  • Em caso de incidente, notificação ao titular em até 72h conforme LGPD Art. 48
